ResearchEnvBench

Benchmarking agents on environment synthesis for research code execution

Home Publication Overview Pyramid Findings Citation

arXiv 2026 · Runtime-first benchmark

ResearchEnvBench studies whether agents can turn a raw research repository into a runnable AI environment.

Existing agent benchmarks usually start after the execution environment already works. ResearchEnvBench moves one step earlier: the agent must resolve dependencies, align CUDA and framework versions, and verify that real entrypoints run under hardware constraints.

Overview

From installation success to real executability.

Why this benchmark

Many agent evaluations assume the execution environment already exists. In real research code, that is exactly the hard part: dependencies are underspecified, hardware stacks are brittle, and repository entrypoints often fail even after “successful” installation.

Task setup

Given a raw research repository, its documentation, and a target execution setting, the agent must use shell execution, file inspection, and auxiliary setup edits to synthesize a runnable environment without modifying tracked source files.

Dataset scope

ResearchEnvBench curates 44 post-2024 repositories spanning modern AI and HPC stacks, covering custom CUDA kernels, distributed training, multimodal systems, and heavyweight research tooling.

Runtime Verification Pyramid

A staged protocol from static integrity to distributed readiness.

C0

Static integrity

Missing imports reported by pyright, used as an auxiliary proxy for dependency closure.

C1

CPU execution

Check whether the repository entrypoint runs under a minimal CPU-only configuration.

C2

CUDA alignment

Verify that framework binaries and drivers are actually aligned, not just installed.

C3

Single-GPU execution

Require real computation on GPU, which exposes hidden ABI and operator mismatches.

C4

Distributed readiness

For supported repos, validate multi-GPU DDP execution as the highest readiness bar.

C5

Hallucination audit

Compare the agent’s self-report against hidden probes to measure false success claims on paths, versions, and claimed capabilities.

Main Findings

What current agents still miss.

GPU-visible is not repository-ready

All evaluated agent settings show a steep drop from CUDA alignment to true entrypoint execution. The paper reports the best multi-GPU validation success rate at only 37.5%.

Static closure is not enough

Lower missing-import rates do not reliably translate into better GPU or DDP readiness. Installable stacks can still fail when the repository touches compiled operators and hardware-sensitive paths.

Failures concentrate in hidden system details

The dominant blockers are native CUDA/C++ extensions, auxiliary tooling such as tracking utilities, and mixed-framework stacks that only break when the real training or inference path is executed.

Agents overclaim readiness

Capability hallucination is a recurring issue: agents infer success from installation logs instead of proving the corresponding runtime probe really passes.

Why it matters

ResearchEnvBench isolates a missing prerequisite for autonomous scientific research: not just editing code, but building an environment where the code can actually run. It turns “environment setup” from an invisible assumption into a measurable capability.

Evaluation Setting

  • Each task runs in an isolated Docker sandbox on Ubuntu 22.04.
  • The paper evaluates repositories under CUDA 12.4 drivers with 2× NVIDIA RTX 4090 GPUs.
  • Agents interact through shell execution, file reading, and auxiliary file editing.
  • Tracked repository source files are not allowed to be modified.
