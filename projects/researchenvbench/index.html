ResearchEnvBench

Benchmarking agents on the missing precondition of research automation: environment synthesis for real code execution.

ResearchEnvBench evaluates whether an agent can transform a raw research repository into an executable environment under realistic hardware and dependency constraints. Instead of assuming the runtime already works, it measures the whole bootstrap process from dependency closure to single-GPU and multi-GPU execution.

01

The gap in current evaluation

Why This Benchmark Exists

Most coding-agent and scientific-discovery benchmarks begin after environment setup has already succeeded. ResearchEnvBench evaluates the step that blocks real research execution first.

Recent coding-agent and scientific-discovery benchmarks usually begin after the execution environment is already available. In real AI research, that assumption is too strong: aligning Python packages, CUDA stacks, compiled operators, and distributed runtimes is often the first failure point.

ResearchEnvBench isolates this missing capability. Given a research repository, its documentation, and a target execution setting, the agent must synthesize an environment that can actually run the repository's entrypoints rather than merely look plausible under shallow checks.

The benchmark asks a more honest question: can an autonomous agent turn a raw research repo into something that really runs?

02

What agents are allowed to do

Benchmark Setup

This is not a toy install benchmark. The setup is designed around modern AI and HPC repositories where environment construction is a systems problem.

The benchmark focuses on modern AI and HPC repositories where environment construction is not a trivial `pip install` problem.

Scope

44 recent repositories

The benchmark selects post-2024 research repos with hardware-sensitive complexity, including custom CUDA kernels and distributed training requirements.

Sandbox

Clean but realistic starting point

Each task begins in a Docker environment with CUDA drivers available but without the repository-specific deep learning stack preinstalled.

Constraint

No tracked source modification

Agents may execute shell commands, inspect files, and add auxiliary setup files, but they are not allowed to edit tracked repository code.

03

How success is measured

Runtime Verification Pyramid

ResearchEnvBench scores environments with a staged runtime protocol instead of relying on installation logs or import checks alone.

Static dependency closure is only the floor. The benchmark keeps climbing until the environment proves it can support real execution under hardware constraints.

C0

Static dependency integrity

Measure missing imports and unresolved packages to test whether the environment is even structurally coherent.

C1

Minimal CPU execution

Require the repository entrypoint to run in a stripped-down CPU setting instead of trusting setup logs.

C2-C4

Hardware-aligned execution

Verify CUDA alignment, single-GPU execution, and distributed readiness as progressively stricter runtime checks.

C5

Capability hallucination audit

Compare the agent's self-report against hidden runtime probes to detect overclaimed success.

04

What current agents still miss

Main Findings

The benchmark shows a persistent gap between apparently valid environments and truly runnable research stacks.

Current state-of-the-art agents still struggle to bridge the gap between environments that look valid and environments that can execute modern research code end to end.

37.5%

Best multi-GPU validation rate

Distributed readiness remains far harder than surface-level setup success.

Steep drop

From CUDA visibility to actual execution

Seeing hardware is not the same as running the repository's real entrypoint.

Hallucination

A first-order reliability issue

Agents often claim success that hidden probes cannot reproduce.

  • Lower missing-import ratios do not guarantee stronger GPU readiness; runtime compatibility is a stricter bottleneck than static closure.
  • Capability hallucination is central to evaluation because environment success is easy to overstate when verification is weak.

05

Why this changes the benchmark target

Why It Matters

ResearchEnvBench turns environment synthesis from an invisible assumption into a first-class evaluation problem for autonomous research agents.

ResearchEnvBench provides a realistic testbed for measuring whether autonomous agents can move from code editing to reproducible scientific execution.

This matters for AI research automation because a repository that cannot be bootstrapped cannot be validated, extended, or reproduced at scale.

If the environment cannot be synthesized, the rest of the research workflow is blocked before any science can happen.
