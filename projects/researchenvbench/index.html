Benchmarking agents on the missing precondition of research automation: environment synthesis for real code execution.

ResearchEnvBench evaluates whether an agent can transform a raw research repository into an executable environment under realistic hardware and dependency constraints. Instead of assuming the runtime already works, it measures the whole bootstrap process from dependency closure to single-GPU and multi-GPU execution.

Yubang Wang, Chenxi Zhang, Bowen Chen, Zezheng Huai, Zihao Dai, Xinchi Chen, Yuxin Wang, Yining Zheng, Jingjing Gong, Xipeng Qiu

Overview

Recent coding-agent and scientific-discovery benchmarks usually begin after the execution environment is already available. In real AI research, that assumption is too strong: aligning Python packages, CUDA stacks, compiled operators, and distributed runtimes is often the first failure point.

ResearchEnvBench isolates this missing capability. Given a research repository, its documentation, and a target execution setting, the agent must synthesize an environment that can actually run the repository's entrypoints rather than merely look plausible under shallow checks.

Benchmark Setup

The benchmark focuses on modern AI and HPC repositories where environment construction is not a trivial `pip install` problem.

  • 44 post-2024 research repositories selected for hardware-sensitive complexity, including custom CUDA kernels and distributed training requirements.
  • A clean Docker sandbox starts with CUDA drivers available but no repository-specific deep learning stack preinstalled.
  • Agents may execute shell commands, inspect files, and add auxiliary setup files, but they are not allowed to modify tracked repository source files.

Runtime Verification Pyramid

ResearchEnvBench scores environments with a staged runtime protocol instead of relying on installation logs or static imports alone.

  • C0: static dependency integrity measured through missing imports.
  • C1: minimal CPU execution of the repository entrypoint.
  • C2-C4: hardware alignment, single-GPU execution, and distributed readiness.
  • C5: capability hallucination, measuring the gap between agent self-reports and hidden runtime probes.

Main Findings

Current state-of-the-art agents still struggle to bridge the gap between apparently valid environments and truly runnable research stacks.

  • All evaluated agents show a steep drop from CUDA visibility to true repository entrypoint execution.
  • The best reported multi-GPU validation success rate is only 37.5%, highlighting how hard distributed readiness remains.
  • Lower missing-import ratios do not guarantee stronger GPU readiness; runtime compatibility is a stricter bottleneck than static closure.
  • Agents frequently overclaim success, making capability hallucination a central reliability issue rather than a side effect.

Why It Matters

ResearchEnvBench turns environment synthesis from an invisible assumption into a first-class benchmark target. It provides a realistic testbed for measuring whether autonomous agents can move from code editing to reproducible scientific execution.

This matters for AI research automation because a repository that cannot be bootstrapped cannot be validated, extended, or reproduced at scale.
